## Notes

Designs live in `templates/Homdutio Pro.html` (`Homdutio Pro.dc.html`). This is a
Design Component: logic in `class Component extends DCLogic { … }`, view in the
`<x-dc>…</x-dc>` template. Styles are inline (objects in `renderVals()`),
visibility driven by `<sc-if>` / `<sc-for>`.

### A. Guiding principle
Notifications are **per-device, not per-account**. Push only reaches the device
that turned it on itself. Activation is possible only on a phone / installed PWA
(Service Worker + Notification API). Desktop never activates — it only informs
and instructs.

### B. State model (per-device)
Anchor: `state = {` (top of the class) — fields:
- `permission: 'default' | 'granted' | 'denied'` — the current device's consent
  (read/write in `localStorage['homdutio_notif_perm']`).
- `notifDevices: [{ id, name, type:'mobile', enabled }]` — the account's other
  devices (seed: "Rafał's iPhone", "Kitchen iPad", both `enabled:false`).
- `sysPromptOpen`, `softAskDismissed`, `denyHelpOpen`, `pushNote`.
- Read consent at startup: anchor `homdutio_notif_perm` in `componentDidMount`.

### C. Device detection
Anchor: getter `get isMobile(){ return this.state.w < 1000; }`. `isMobile` = phone
(supports notifications); otherwise desktop (unsupported). `this.state.w` is
updated in `componentDidMount` (resize listener).

### D. Activation only on mobile (soft-ask → system prompt)
Anchor: `requestNotifs()` — if `!isMobile` → return (desktop does not activate);
otherwise open a simulated system prompt (`sysPromptOpen:true`).
- Simulated OS prompt: in the template `<sc-if value="{{ sysPromptOpen }}">`,
  buttons Don't Allow / Allow (tokens `{{ sysDeny }}` / `{{ sysAllow }}`).
- Anchor `sysAllow()` → `permission:'granted'`, write to localStorage, close,
  toast "Notifications on for this device".
- Anchor `sysDeny()` → `permission:'denied'`, open unblock instructions
  (`denyHelpOpen:true`). Never call the system prompt without a user click.

### E. Soft-ask (banner) — mobile only
Visibility anchor: `const showSoftAsk = isMobile && permission !== 'granted' && !S.softAskDismissed;`
Template: `<sc-if value="{{ showSoftAsk }}">` (in the Board view, above the columns).
- Content depends on `softAskDenied` (`permission === 'denied'`): normally
  "Turn on notifications?" + CTA "Enable notifications" (`{{ softAskAction }}` →
  `requestNotifs`); after denial "Notifications are blocked" + CTA "How to
  unblock" (→ `openDenyHelp`).
- Dismiss: `{{ dismissSoftAsk }}` (session-scoped — the soft-ask may return, this
  is not "never again").
- Post-denial instructions: `<sc-if value="{{ denyHelpOpen }}">` (steps to unblock
  in the browser).

### F. Desktop — no "Enable" button, only status + instructions + QR
- Board banner: anchor `const showDeskBanner = !isMobile && !anyEnabled && !S.softAskDismissed;`
  → template `<sc-if value="{{ showDeskBanner }}">` ("Get notified on your phone",
  no activation CTA).
- In Settings: `showDesktopInstruct: !isMobile` → template
  `<sc-if value="{{ showDesktopInstruct }}">` (a "Turn on from your phone" frame +
  QR code). The QR code is generated by `qrEl(px)` (SVG), exposed as
  `qrCode: this.qrEl(150)`.
- No toggle/enable button of any kind on desktop.

### G. Settings = management, not activation
- Device list: anchor `const deviceRows = deviceList.map(...)` → template
  `<sc-for list="{{ deviceRows }}" as="d">`. Per-device status: On / Off (badge).
  Current device marked THIS DEVICE.
- Account status: `notifStatusText` (e.g. "Notifications aren't on for any of your
  devices yet." or "Notifications are on for …").
- Current device on the list only when supported (phone). Anchor:
  `const deviceList = isMobile ? [currentDevice, ...otherDevices] : otherDevices;`
  — on desktop the dead "This computer / Not supported" row is omitted.
- The only activation button in Settings exists solely for the current phone when
  not yet enabled: `d.showEnable` → `{{ enableHere }}` (→ `requestNotifs`, i.e.
  soft-ask/system prompt again, not a persistent toggle).
- PWA hint (mobile, when no consent): `showInstallHint: isMobile && permission !== 'granted'`
  → "Install Homdutio… so notifications work like a native app."

### H. Real push delivery
Anchor: `pushNotify(title, body)` — works only when
`this.isMobile && this.state.permission === 'granted'` (push lands only on a
device that has it enabled). Render the notification toast: template
`<sc-if value="{{ pushNote }}">`.

### I. Assigning people to tasks
- Only an admin can assign: anchor `showAssign: meIsAdmin` in `renderVals`;
  person-selection chips: `const assignChips = [...]` → template
  `<sc-for list="{{ assignChips }}" as="a">` in the add/edit modal ("Anyone" =
  no assignment).
- Save logic: anchor `submitTask()`. On assignment: if status `todo` → moves to
  `progress`, `claimer = assignee`. Notification depends on the recipient:
  - `assignee === this.meName` → `pushNotify('New task assigned to you', …)`
    (reaches me only if MY device has it enabled).
  - other recipient → `this.flash(assignee + ' will be notified on any device
    where they've turned notifications on.')` (reflects the per-device model — the
    assigner does not know/decide the recipient's consent).
- Recipient flow (the target of the soft-ask/desk banner): opens their dashboard →
  if no consent, on a phone they see the soft-ask (section E), on desktop a banner
  telling them to enable it on their phone (section F).
